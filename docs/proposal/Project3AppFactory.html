Project Proposal - App Factory

1.0 Concept

Our project is an application for app ideas. We will utilise OAuth to authenticate users with one of two statuses; standard user and developers. All users are able to post their ideas for a new web application that satisfies a particular business or personal need.

Once the app idea has been created, it can be viewed by all other users on the App Factory homepage. Developers can choose to mark themselves as ‘building’ the application and therefore working on a solution. Using GitHub OAuth, each developer working on a solution can link a repo to their solution to enable the poster and any other users to view a developer’s progress. Developers will also have the facility to make comments and post links to deployed applications in the solutions page.

This application allows normal people with good ideas for new and original web applications to have their ideas realised. It also allows developers, with a range of experience, to test themselves and build applications which satisfy a real-world need.

2.0 User Stories 

AS a person who is passionate about technology, I WANT an app where I can find the latest applications SO THAT I can be up to date.

AS a creative individual, I WANT an app where I can post my ideas about applications SO THAT other people can vote on and developers who appreciate the idea to make it possible.

AS a person who is passionate about technology, I WANT an app where I can keep track of all my ideas SO THAT I can quickly view my dashboard status.

AS a person who is passionate about technology, I WANT an app where I can see which developers are working on my posted ideas SO THAT I can quickly check the progress.

AS a developer, I WANT an app that represents a pool of interesting ideas SO THAT I can help in creating good apps that people can enjoy and use.

AS a developer, I WANT an app where I can view other developers' work on those ideas SO THAT I can have my input and opinions communicated in an integrated platform.

3.0 Application functionality

3.1 Authentication

MVP: Authentication with Google OAuth and GitHub (for developers) which will utilise Passport.js and Google OAuth 2.0. The user will specify whether they are solely a poster or a developer/poster, which will dictate the status of the user which will inform what components will be rendered in their dashboard. User information will be stored to MongoDB using the Mongoose ORM.

Additional  features: Additional OAuth using the Facebook OAuth package. Implement Git into the application to allow for better integration between app ideas and developers’ projects.

3.2 Post new app idea

MVP: Capture app idea on a front end form and, upon submission, data is validated and sent to bespoke API.

Additional features: Implement text editor, allowing the user to save formatted text to DB. Addition of an attachment facility to enable the user to attach any additional documents which would help an interested developer in building the app e.g. design concepts.

3.3 View all posted app ideas

MVP: This will be the homepage wherein the user will be able to view a paginated listing of all posted ideas. Ideas will be scored via a voting system where registered users can vote up or down app ideas. Each app idea will display the title, category, technologies required (where applicable), a short summary, poster name (icon to denote standard poster or developer), post date and the score.

Additional features: Incorporate badges which are assigned to users for posting a certain number of app ideas and completed ideas/votes.

3.4 Sort posts

MVP: By default, app ideas will be sorted by score, but users will be able to re-sort based on category or post date. This will be done via a dropdown box.

3.5 Filter post

MVP: The user will be able to filter by a specific category or categories, date range, score threshold, poster type and technologies required (where applicable).

3.6 Voting

MVP: Each app idea on the homepage will have a thumbs up and thumbs down icon, which can be clicked by the user to up or down vote that particular post. The total vote will then be displayed against each app idea.

Additional features: The vote will be hidden for the first 24 hours.

3.7 View individual post

MVP: Users will be able to click on a single app idea from the homepage and they will be directed to a new page displaying the full details of the specific idea/concept that was clicked. The details shown will be those that were submitted by the original poster, retrieved from the DB using Mongoose. The app idea will have a comments section as well as a section for solutions, which will also implement a voting system.

3.8 Developer to mark themselves as working on a solution

MVP: Users logged into the application as a developer will be able to click a button which will mark them down as currently working on the project. These would then be displayed in a box alongside the app idea. The developer will be required to create a GitHub repo with the solution title which will then be used to retrieve the repo

3.9 Dashboard

MVP: Standard users will have a single tab displaying the app ideas that they have posted whereas developers will have an additional tab displaying the solutions they are currently working on (see below). The list of app ideas will be a duplicate of the list on the homepage with app ideas filtered to only show those posted by the current user.

Additional features: Statistics page showing various breakdowns of the progress of app ideas.

3.10 View all apps being worked on by developer (dashboard)

MVP: The list of solutions will be a duplicate of the list on the homepage with solutions filtered to only show those being worked on by the current user (developer).

3.11 Developer solution updates

VP: From the dashboard, the developer will be able to select a single post where they will be able to add comments outlining the progress of the project and submit links to the deployed application.

Additional feature: Open issue through comments section of solution.

3.12 Offline capability

MVP:

We will be using create-react-app and initializing the pre-built service worker to enable basic offline capability.

Additional features: Deferred app postings while offline and push notifications to indicate

3.13 Server

MVP: We will be building a Content Management System (CMS) that stores, manages and delivers content without a front-end delivery layer being required. This will allow more fluidity on the front-end as we will not be restricted by back-end structure.

3.14 Back-end Routes

MVP: The backend routes being implemented in our project will be split into two files. The first file will be api.js which will include endpoints to achieve the following functionality: post new idea, get all ideas and get single idea. The second file will be auth.js, which will include routes for our OAuth redirect URIs to ensure authentication of users.

3.15 Database

MVP: Data pertaining to users and app ideas will be saved to separate collections in MongoDB using Mongoose. The structure of these collections will be as follows:

3.16 Ideas for further development

Testing system

Users can test their proficiency in different languages and be awarded badges to denote their skill.

Difficulty rating

Developers to complete a survey indicating how difficult they found the project on a scale of 1-10 - this would then be calculated (utilising developer skill) and averaged to create a score  and assigned to the app idea.

Fun v Business

Split the application into two sections where users can post ideas for fun or for business purposes where developers would be paid for their contribution to a project.
